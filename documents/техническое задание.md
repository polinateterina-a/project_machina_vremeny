# ТЕХНИЧЕСКОЕ ЗАДАНИЕ

## 1. Цель

Разработать интерактивную веб-платформу «Перекрёсток эпох», которая позволяет пользователю совершить аудиовизуальное путешествие по дореволюционному Красноярску через сопоставление исторических и современных фотографий с одновременным прослушиванием реконструированных звуковых ландшафтов.

**Ключевые задачи ТЗ:**

- Реализовать веб-приложение с интерактивной картой, содержащей 4 локации пешеходного маршрута.
- Обеспечить возможность поиска экскурсии и переключения между режимами отображения (современный Красноярск ↔ дореволюционный Красноярск).
- Создать для каждой локации карточку с двумя фотографиями («было/стало»), аудиоплеером и историческим описанием.
- Обеспечить удобство использования.

## 2. Описание информационной системы

Платформа «Перекрёсток эпох» — это веб-приложение с клиент-серверной архитектурой, состоящее из:

| Компонент       | Описание                                                                 |
|----------------|--------------------------------------------------------------------------|
| **Фронтенд**    | Веб-интерфейс на HTML/CSS/JS, отображающий карту, поиск, карточки локаций и аудиоплеер |
| **Бэкенд**      | Сервер на Python/Flask, обрабатывающий API-запросы, работающий с базой данных и файловым хранилищем |
| **База данных** | SQLite, хранящая информацию о городах, экскурсиях, локациях              |
| **Файловое хранилище** | Директория на сервере для хранения фотографий и аудиофайлов               |

### Модули системы

| Модуль                     | Назначение                                                       |
|----------------------------|------------------------------------------------------------------|
| **Модуль карты**           | Отображение интерактивной карты с метками локаций                |
| **Модуль поиска**          | Поиск экскурсий по названию и описанию                           |
| **Модуль экскурсии**       | Отображение списка локаций, входящих в экскурсию                 |
| **Модуль карточки локации**| Отображение фото, слайдера «было/стало», аудиоплеера и текста    |
| **Модуль переключения времени** | Переключение отображения карты между современным и дореволюционным состоянием |
| **Административный модуль**| Добавление и редактирование экскурсий и локаций (в MVP — через админ-панель) |

### Объекты системы

| Объект         | Атрибуты                                                          |
|----------------|-------------------------------------------------------------------|
| **Город**      | id, название, описание                                            |
| **Экскурсия**  | id, название города, описание, список локаций                     |
| **Локация**    | id, название, координаты (lat/lng), историческое фото, современное фото, аудиофайл, описание, год, теги |
| **Пользователь**| (в MVP не требуется регистрация, только сессия)                   |

## 3. Функциональные требования

### Модуль карты

1. Приложение должно отображать интерактивную карту на базе OpenStreetMap.
2. Карта должна иметь возможность приближения/отдаления (зум) и перемещения (драг-н-дроп).
3. На карте должны быть размещены маркеры всех локаций, входящих в экскурсию.
4. При клике на маркер должна открываться карточка с краткой информацией о локации и кнопкой «Подробнее».
5. Должна быть возможность переключения отображения карты между современным Красноярском и дореволюционным Красноярском (смена карт или наложение исторического слоя).

### Модуль поиска экскурсий

1. На главной странице должно быть поле поиска.
2. Пользователь должен иметь возможность искать экскурсию по названию (например, «Красноярск»).
3. Поиск должен работать по названию экскурсии, названиям локаций и тегам.
4. При выборе экскурсии из результатов поиска должен открываться её маршрут на карте.
5. Если экскурсия не найдена, должно отображаться сообщение «Ничего не найдено».

### Модуль экскурсии

1. Экскурсия должна включать 4 локации, связанных пешеходным маршрутом.
2. Локации должны отображаться на карте в порядке маршрута.
3. Должна отображаться общая информация об экскурсии (название, описание, протяжённость, время прохождения).

### Модуль карточки локации

1. При клике на точку должно открываться модальное окно с полной информацией о локации.
2. Модальное окно должно содержать слайдер «Было/Стало» для переключения между историческим и современным фото.
3. Модальное окно должно содержать аудиоплеер для воспроизведения звуковой реконструкции.
4. Модальное окно должно содержать историческое описание локации (200–300 слов).
5. Модальное окно должно содержать год постройки/период и теги.
6. Модальное окно должно закрываться по кнопке «X» или клику вне окна.

### Модуль аудиоплеера

1. Аудиоплеер должен поддерживать воспроизведение и паузу.
2. Аудиоплеер должен отображать текущее время воспроизведения и общую длительность.
3. Должна быть возможность перемотки.
4. При закрытии модального окна воспроизведение аудио должно автоматически останавливаться.
5. Аудиофайлы должны быть в формате MP3.

### Административный модуль (MVP)

1. Администратор должен иметь доступ к админ-панели по отдельному URL.
2. Администратор должен иметь возможность добавлять новые экскурсии.
3. Администратор должен иметь возможность добавлять новые локации в экскурсию.
4. Администратор должен иметь возможность редактировать существующие локации.
5. Для загрузки фото и аудио используется форма с указанием пути к файлам (в MVP — ручное размещение файлов на сервере).

## 4. Нефункциональные требования

### Производительность

1. Время загрузки главной страницы не должно превышать 2 секунд при стабильной скорости интернета.
2. Время отклика API на запрос списка локаций не должно превышать 500 мс.
3. Карта должна загружаться и становиться интерактивной не более чем за 3 секунды.
4. Аудиофайлы должны начинать воспроизводиться не более чем через 1 секунду после нажатия «Play».

### Доступность и адаптивность

1. Интерфейс должен корректно отображаться на разрешениях от 320px до 1920px.
2. Платформа должна поддерживаться в браузерах: Яндекс, Chrome, Firefox, Safari, Edge.
3. Мобильная версия должна иметь удобные элементы управления для пальцевого ввода.
4. Шрифты должны быть читаемыми на всех устройствах (минимальный размер 14px для основного текста).

### Надёжность и безопасность

1. При ошибке загрузки данных должно отображаться понятное пользователю сообщение.
2. Система должна корректно обрабатывать отсутствие файлов (фото, аудио) — отображать заглушку.
3. Админ-панель не должна быть доступна обычным пользователям.
4. Все пользовательские данные передаются по протоколу HTTPS.

### Удобство использования

1. Интерфейс должен быть интуитивно понятным, не требовать обучения.
2. Кнопки должны иметь понятные иконки или подписи.
3. Маркеры на карте должны быть визуально различимы.
4. Слайдер «Было/Стало» должен иметь визуальный эффект переключения.

### Технические требования

1. Бэкенд реализован на Python 3.10+ с использованием Flask.
2. База данных — SQLite.
3. Код должен храниться в репозитории на GitHub.
4. Должен быть файл `requirements.txt` со всеми зависимостями.
5. Должна быть инструкция по развёртыванию.

## 5. Критерии приёмки

### Функциональные критерии

1. Пользователь может открыть страницу экскурсии и увидеть карту с 4 маркерами.
2. Пользователь может найти экскурсию «Красноярск» через поиск.
3. Пользователь может нажать на маркер и увидеть краткую информацию о локации.
4. Пользователь может нажать «Подробнее» и открыть модальное окно с полной информацией.
5. В модальном окне работает слайдер «Было/Стало» (переключение между двумя фото).
6. В модальном окне работает аудиоплеер (воспроизведение, пауза, перемотка).
7. При закрытии модального окна аудио останавливается.
8. Пользователь может переключить карту в режим «дореволюционный Красноярск».
9. Все 7 локаций имеют историческое фото, современное фото, аудио и описание.
10. Порядок локаций на карте соответствует пешеходному маршруту.

### Технические критерии

1. API возвращает корректные JSON-данные для всех эндпоинтов.
2. База данных содержит записи для 1 города, 1 экскурсии, 4 локаций.
3. Все статические файлы (фото, аудио) доступны по указанным путям.
4. Приложение запускается одной командой и работает без ошибок.
5. Код проходит проверку линтером (без критических ошибок).

### Пользовательские критерии

1. Пользователь может пройти весь маршрут из 4 локаций без технических сбоев.
2. Пользователь понимает, как пользоваться картой, поиском и плеером без инструкции.
3. Время загрузки между локациями не превышает 3 секунд.

## 6. DEFINITION OF DONE

**DoD для проекта в целом**

1. Все функциональные требования (раздел 3) реализованы и работают.
2. Все нефункциональные требования (раздел 4) выполнены.
3. Все критерии приёмки (раздел 5) пройдены.
4. Проведено тестирование на реальной фокус-группе (не менее 30 человек).
5. Получены положительные отзывы от не менее 70% тестировщиков.
6. Код загружен в репозиторий GitHub с историей коммитов.
7. Написана инструкция по развёртыванию и запуску.
8. Проект развёрнут на тестовом сервере и доступен по ссылке.
9. Отсутствуют критические баги, влияющие на работу пользователя.
10. Проведена презентация проекта.
